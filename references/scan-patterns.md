# 安全语义审查与误报鉴别指南

`audit_scanner.py` 输出 JSON 结果后，你需要作为安全专家鉴别这些发现，并对项目代码开展语义层面的行为审查。

## 一、 鉴别脚本扫描结果的“误报”（False Positives）

脚本采用正则表达式进行机械匹配，容易产生噪音。请利用你的上下文理解能力过滤误报：

### 🔴 一级风险：凭据泄露（API Key, Password, Token）
- **真阳性 (报告它)**：
  - `const API_KEY = "sk-ant-api03-xxxxxxxxx"`
  - `password: "real_db_password_123!"`
  - `postgres://admin:secret123@localhost:5432/db`
- **假阳性 (忽略它)**：
  - 出现在字典 Key 中：`{ "password": user_input }`
  - 占位符或示例：`sk-xxxxxxxxxxxx`、`YOUR_API_KEY_HERE`
  - 测试用例中的假数据：`mock_token = "123456789"`

### 🟡 二级风险：个人信息（Email, IP, 路径）
- **真阳性 (报告它)**：
  - 真实的个人邮箱（如包含姓名拼音的邮箱）、内网 IP (`10.x.x.x`)。
  - 本机绝对路径泄露（如 `/Users/wangermazi/Desktop/my_secret_project/...`），暴露了系统用户名。
- **假阳性 (忽略它)**：
  - 开源协议中的 `author@example.com`
  - 默认的 `127.0.0.1` 绑定地址

## 二、 代码行为的“使用者安全”审查

此类审查无法由脚本完成，需要你主动阅读核心代码。目的不是寻找 Bug，而是保护运行此代码的接收方安全：

1. **未经声明的网络出站请求**
   - 检查点：代码中是否存在隐蔽的 `requests.post()`、`curl` 或前端 `fetch()` 发送数据到外部服务器？
   - 评估：天气插件发送请求属正常行为；但本地数据处理工具若发送网络请求，则须在报告中警告。
2. **破坏性的文件操作**
   - 检查点：代码是否存在 `os.remove()` 或文件覆盖写入？
   - 评估：是否存在意外删除用户文件的风险？若有，应在 README 中披露。
3. **动态代码执行**
   - 检查点：`eval()`、`exec()`、`subprocess.run()` 且参数包含外部输入。
   - 评估：存在严重的注入风险。
4. **权限行为**
   - 检查点：脚本是否强制要求 `sudo`，或修改了 `/etc`、用户 `~/.config`？
   - 评估：须在报告的“使用者安全警告”部分明确指出。

## 三、 合规检查解读

脚本会检测是否缺少 `README`、`LICENSE` 或 `.gitignore`。
- 如果缺失，请在报告结尾提供合规建议。
- 主动分析现有的 `README` 是否包含“安装步骤”和“环境要求”。
