# 安全语义审查与误报鉴别指南

当 `audit_scanner.py` 吐出 JSON 结果后，你需要像一个真正的安全专家一样对结果进行鉴别。同时，你还需要对项目进行语义上的行为审查。

## 一、 如何鉴别脚本扫描结果的“误报”（False Positives）

脚本使用的是基于正则表达式的机械匹配，它会产生很多噪音。你需要利用大语言模型（你）的上下文理解能力进行过滤：

### 🔴 一级风险：凭据泄露（API Key, Password, Token）
- **真阳性 (报告它)**：
  - `const API_KEY = "sk-ant-api03-xxxxxxxxx"`
  - `password: "real_db_password_123!"`
  - `postgres://admin:secret123@localhost:5432/db`
- **假阳性 (忽略它)**：
  - 出现在字典 Key 中：`{ "password": user_input }`
  - 占位符或示例：`sk-xxxxxxxxxxxx`、`YOUR_API_KEY_HERE`
  - 测试用例中的假数据：`mock_token = "123456789"`

### 🟡 二级风险：个人信息（Email, IP, 路径）
- **真阳性 (报告它)**：
  - 真实的个人邮箱（如用户名的拼音邮箱）、内网环境 IP (`10.x.x.x`)。
  - 本机绝对路径泄露（如 `/Users/jalyn/Desktop/my_secret_project/...`），这会暴露使用者的系统用户名。
- **假阳性 (忽略它)**：
  - 开源协议中的 `author@example.com`
  - 默认的 `127.0.0.1` 绑定地址

## 二、 代码行为的“使用者安全”审查

这类审查脚本无法完成，需要你主动去查看核心代码文件。这不是为了找 Bug，而是为了保护**将要运行这份代码的接收方**：

1. **未经声明的网络出站请求**
   - 检查点：代码中是否有隐蔽的 `requests.post()`、`curl` 或前端的 `fetch()` 把数据发往了外部服务器？
   - 评估：如果这是一个天气插件，发请求是正常的。但如果是本地数据处理工具，却在发网络请求，这就需要在报告中警告。
2. **破坏性的文件操作**
   - 检查点：代码是否有 `os.remove()`、覆写文件等操作？
   - 评估：是否可能意外删除使用者的文件？建议在 README 中披露。
3. **动态代码执行**
   - 检查点：`eval()`、`exec()`、`subprocess.run()` 且参数包含外部输入。
   - 评估：存在严重的注入风险。
4. **权限行为**
   - 检查点：脚本是否强行要求 `sudo`，是否修改 `/etc` 或用户的 `~/.config`？
   - 评估：必须在报告的「使用者安全警告」中明示。

## 三、 合规检查解读

脚本会告诉你是否缺少 `README`、`LICENSE` 或 `.gitignore`。
- 如果缺失，在报告的最后给出合规建议。
- 你可以主动分析现有的 `README` 是否写明了“安装步骤”和“环境要求”。
