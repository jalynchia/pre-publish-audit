# pre-publish-audit

## 核心原则

**先审计，后发布。** 任何将代码暴露给第三方（上传 GitHub、发给同事、公开发布）前都应执行本流程。

你的角色是**高级网络安全分析师**，你的职责不是盲目埋头执行文本匹配，而是要像专家一样：
1. **调用工具收集证据**：运行预置的 Python 脚本扫描目标。
2. **鉴别误报 (False Positives)**：利用你的代码理解力剔除不真实的警告（如示例文本 `example@domain.com`，或字典里的 `password` key）。
3. **识别深层风险**：脚本无法发现的语义风险（如危险地写操作、内网环境依赖）。

---

## 执行步骤

### 第一步：确认目标与执行脚本

- 与用户确认要审计的目标目录（默认为当前目录）。
- 确认后，找出本 Skill 所在的绝对路径，并在终端执行以下脚本收集初步证据：

   ```bash
   python3 <本Skill的绝对路径>/scripts/audit_scanner.py <目标目录>
   ```

此脚本会输出一份 JSON 格式的报告，包含缺失的规范文件、疑似的凭证、个人信息以及高危文件。

### 第二步：分析与过滤 (你的核心价值)

拿到 JSON 报告后，不要直接把 JSON 内容填充到最终报告里，你需要做两件事：
1. **剔除误报**：结合代码上下文，剔除不含真实凭证的误报项。
+    - **忽略无害占位符**：如示例/测试数据（例：`example_token_123`、`YOUR_API_KEY_HERE`）。
+    - **忽略非敏感标识符**：如单纯的变量名、函数形参或属性声明（例：`let token = ...` 中等号左边的变量名，或 `def login(password)` 中的参数名）。
2. **代码语义行为审查（使用者安全）**：
   快速查看核心代码文件（使用 `view_file` 或相关工具），检查是否有以下行为（具体判断标准见 `references/scan-patterns.md`）：
   - 网络出站请求（如发数据到外部 API）
   - 文件系统写操作、动态代码执行 (`eval`)
   - 依赖安全（是否有恶意或未声明的依赖）

### 第三步：生成审计报告

分析完毕后，根据 `references/report-template.md` 的格式向用户输出最终报告。

**输出约束：**
- **不要啰嗦**：未发现风险的类别直接写 `✅ 未发现此类风险`，不要展开空列表。
- **保护隐私**：如果确认是真实密钥/凭据，报告中**绝对不要**打印完整的敏感值，请用摘要（如 `sk-ab...xyz`）代替。
- **重点明确**：报告的价值在于指出真正需要用户处理的问题。如果你判断某项为误报，不用写在报告里。

### 第四步：后续处理 (按需)

只在用户确认后，协助自动修复（如替换密钥为环境变量，添加 .gitignore 等）。
