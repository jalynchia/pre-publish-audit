# 审计报告格式模板

## 使用规则

- 报告直接输出到对话，让用户直观阅读
- 未发现风险的章节，用一行 `✅ 未发现风险` 代替，不展开空列表
- **不要**将完整的 API Key、密码或 JWT 贴入报告——用摘要片段描述即可（如 `sk-ab...xy`）
- 不确定是否为真实凭据的匹配项，标记为「待确认」交用户判断

---

## 报告结构

```
# 📋 发布前安全审计报告

审计目标：<路径>
审计时间：<时间>
文件总数：<数量>
扫描层级：四层完整扫描

---

## 🔴 一级风险（必须处理，否则不建议发布）

### 风险 1：[类别] 具体发现
- 文件：<路径:行号>
- 内容摘要：<敏感内容的上下文片段，不展示完整敏感值>
- 风险说明：<为什么是问题>
- 处理建议：<采用环境变量 / 替换为占位符 / 移除等具体方式>

### 风险 2：...

✅ 未发现一级风险（如无）

---

## 🟡 二/三/四级风险（建议处理）

### 建议 1：[类别] 具体发现
- 文件：<路径:行号>
- 内容摘要：<描述>
- 建议：<具体建议>

✅ 未发现建议级风险（如无）

---

## ⚠️ 使用者安全警告

### 警告 1：[类别，如「网络出站请求」]
- 发现位置：<文件:行号>
- 行为描述：<代码在做什么>
- 对使用者的影响：<可能影响>
- 建议：<如何在 README 中披露 / 如何处理>

✅ 未发现使用者安全警告（如无）

---

## 💡 合规与完整性建议

- ❌ 缺少 README.md：建议创建，包含环境要求 / 安装步骤 / 使用示例 / 安全注意事项
- ❌ 缺少 LICENSE：建议选择 MIT 许可证（或根据项目性质推荐其他）
- ⚠️ 缺少 .gitignore：建议添加（可在此提供推荐模板）
- ⚠️ 缺少依赖声明文件

✅ 合规检查全部通过（如无问题）

---

## 📝 总结与下一步

本次审计共发现 X 项一级风险、Y 项建议级问题、Z 项使用者安全警告、W 项合规建议。

请确认：
1. 是否同意上述风险评估？
2. 哪些风险需要我帮助自动处理？
3. 哪些由您手动处理？
```

---

## 自动处理参考（用户确认后才执行）

| 风险类型 | 自动处理方式 |
| --- | --- |
| 凭据泄露（API Key、密码） | 替换为 `os.environ.get("变量名")` 或 `<YOUR_API_KEY_HERE>` 占位符 |
| 本机绝对路径泄露 | 替换为 `os.path.expanduser("~")` 或示例路径 |
| 个人信息（邮箱、手机号） | 替换为 `your@email.com` 等示例占位符 |

## 无法自动处理的项（提供指引）

| 项目 | 处理指引 |
| --- | --- |
| EXIF 数据清理 | `exiftool -all= <文件>` 批量清理；或 macOS 预览「工具 → 显示检查器 → GPS」手动移除 |
| 许可证选择 | 提供 MIT / Apache 2.0 / GPL 对比表，让用户决定 |
| `.git` 历史泄露 | 建议新建仓库而非清理历史（历史清理易出错且不彻底） |
